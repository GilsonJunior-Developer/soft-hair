## Passo 1 — Confirmação de booking pelo fluxo público

### 1.1. Abrir o link de booking

Cole no browser:
```
https://soft-hair-web-git-feat-ab7387-gilsonjunior-developers-projects.vercel.app/salao-teste-prod1/maria-teste/book
```

**Esperado:**
- Página carrega normalmente (não 404, não 500)
- Aparece "Agendar com Maria Teste" no topo
- Lista de serviços aparece como cards clicáveis

✅ ☐ Booking page carregou OK

### 1.2. Selecionar serviço

Clique em qualquer serviço da lista. Você deve avançar automaticamente pro passo 2.

**Esperado:**
- URL muda pra incluir `?s=<uuid>` (mostra que o serviço foi selecionado e está shareable)
- Step indicator no topo destaca "Horário"

✅ ☐ Service step avançou

### 1.3. Escolher data e horário

- Aparece uma faixa horizontal de 14 dias na parte de cima
- Ao clicar em um dia (que tenha slots), aparece grid de horários
- **Sugestão:** escolha um slot pelo menos **48 horas no futuro** (se for muito perto, o teste de cancel pode falhar pelo cancel window de 24h)

Clique em um horário disponível.

**Esperado:**
- Step indicator destaca "Confirmar"
- Aparece resumo do serviço + horário escolhido
- Form de contato aparece

✅ ☐ DateTime step avançou

### 1.4. Preencher contato + LGPD

Use os dados de teste da tabela acima. Marque o checkbox LGPD.

Clique em **"Confirmar agendamento"**.

**Esperado:**
- Botão muda pra "Confirmando…" por ~1-2s
- Browser **redireciona automaticamente** para uma URL longa do tipo:
  ```
  /agendamento/eyJhbGciOiJIUzI1NiIs........<token JWT longo>........?justCreated=1
  ```

⚠️ **CRÍTICO:** verifique que o redirect foi pra `/agendamento/{algo longo}` e **NÃO** pra `/book/success/...`. Esse é o ponto-chave da Story 2.7.

✅ ☐ Redirect foi pra `/agendamento/{jwt}` (não pra success page antiga)

📋 **COPIE A URL DESTA PÁGINA** e cole no fim deste documento (ou no chat) — vamos usá-la nos próximos passos.

## Passo 5 — Testar cancelamento

Volte na aba do **Passo 2** (manage page do booking que você criou).

5.1. Clique em **"Cancelar agendamento"**.

**Esperado:**
- O botão é substituído por um diálogo inline com texto "Tem certeza que quer cancelar este agendamento?" + dois botões: **"Sim, cancelar"** (vermelho) e **"Voltar"** (cinza)

✅ ☐ Confirm dialog apareceu

5.2. Clique em **"Sim, cancelar"**.

**Esperado:**
- Botão muda pra "Cancelando…"
- Após ~1s, aparece bloco verde de status: "✓ Agendamento cancelado" + "Que tal remarcar num horário melhor?"
- Recarregue a página com F5 — o "Status" no resumo agora deve mostrar **"Cancelado"** e os botões "Reagendar"/"Cancelar" desaparecem (substituídos por nota explicando que o agendamento já está finalizado)

✅ ☐ Cancelamento funcionou + status atualizou pra Cancelado

## Passo 6 — Testar reagendamento (opcional mas recomendado)

Volte ao Passo 1 e crie **outro** booking (use os mesmos dados — vai upsertar o mesmo cliente, ou use email/phone diferentes pra criar novo).

Quando estiver na manage page do novo booking:

6.1. Clique em **"Reagendar"**.

**Esperado:**
- Você é redirecionado pra `/agendamento/<jwt>/reagendar`
- Aparece o título "Reagendar com Maria Teste" + dados do serviço atual
- O slot picker carrega (faixa de dias + grid de horários)

✅ ☐ Reschedule page carregou

6.2. Selecione um novo dia e horário diferente do anterior.

**Esperado:**
- Aparece um banner azul claro: "Novo horário: <data formatada por extenso>" + serviço + duração

✅ ☐ Slot selecionado

6.3. Clique em **"Confirmar novo horário"**.

**Esperado:**
- Botão "Confirmando…"
- Redireciona pra **NOVA URL** do tipo `/agendamento/{jwt-DIFERENTE-do-original}?justCreated=1`
- Manage page mostra o novo horário

⚠️ **Importante:** o JWT da URL final deve ser **diferente** do JWT do booking original. Isso prova que a Story 2.7 emite tokens novos a cada reschedule (segurança — token antigo vira inútil).

✅ ☐ Novo JWT emitido (diferente do original)

## Se algo falhar

| O quê | Provável causa | O quê fazer |
|---|---|---|
| Submit do booking dá erro "Servidor não está configurado pra tokens" | `APPOINTMENT_TOKEN_SECRET` não foi pego pelo deploy atual | No Vercel dashboard, force um redeploy do branch `feat/story-2.7-client-appointment-management` (Three-dot menu → Redeploy → use existing build cache OFF) |
| Submit dá "Dados inválidos" ou similar | Bug não pego — voltar pro Dex | Copie a mensagem exata + tire screenshot + sinalize "@dev *apply-qa-fixes 2.7" |
| Manage page dá 404 ou 500 | Rota não foi deployada | Verificar build logs da PR #26 — pode ser cache stale do Vercel |
| Email log mostra `/book/success/` ainda | Algum import stale do código antigo | Voltar pro Dex pra investigar |
