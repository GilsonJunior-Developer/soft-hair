## Story

**As a** dono do salão,
**I want** configurar regras de comissão flexíveis por profissional (% fixa OU tabela por serviço, com override opcional ao nível de serviço) e simular cálculos antes de aplicar,
**so that** o cálculo automático que Story 4.2 vai disparar reflita corretamente os contratos diferenciados, **sem afetar comissões de atendimentos já registrados** (histórico imutável é pré-requisito do relatório mensal de Story 4.3).

## Acceptance Criteria

> Copiados verbatim de `docs/prd/epic-4-finance-commission-nfs-e.md` linhas 11–18.

1. Perfil do profissional suporta 2 modos: (a) % fixa sobre serviço; (b) tabela por serviço (overrides específicos)
2. Serviço pode ter comissão específica que sobrescreve a default do profissional
3. Validação: total (% profissional + % salão) = 100%
4. Simulador interativo: "Se eu fizer 10 cortes a R$ 50, profissional X ganha R$ Y"
5. Alteração de regra NÃO afeta comissões de atendimentos já realizados (histórico preservado)
6. Passa unit test extenso do engine de comissão

### AC1(b) TABLE mode — RESOLVED (Interpretation A approved by @po 2026-05-02)

PRD AC1: _"Perfil do profissional suporta 2 modos: (a) % fixa sobre serviço; (b) **tabela por serviço (overrides específicos)**"_

**Decisão @po:** Interpretation A — nova tabela `professional_service_commissions(professional_id, service_id, percent)` permite overrides per-prof × per-service. Rationale: (i) leitura literal "overrides **específicos**" implica per-prof; (ii) Story 1.5 já expôs o radio TABLE publicamente; (iii) Interpretation B (reuse de service overrides) tornaria mode TABLE redundante com PERCENT_FIXED + AC2.

**Caveat de UX (mandatório, não-opcional) — Task 3.1:** progressive disclosure + bulk-set helper. Owner com 5 profissionais × 20 serviços = 100 inputs potenciais — esse design exigiria mitigação ou contradiz positioning "simples vs Trinks". Soluções aplicadas no Task 3.1: collapsed-by-default; bulk "Aplicar X% a todos" para o caso típico; entries vazios = fallback (não obriga preencher tudo).

### AC3 Sum=100% — RESOLVED (SM interpretation approved by @po 2026-05-02)

PRD AC3: _"Validação: total (% profissional + % salão) = 100%"_

**Decisão @po:** SM interpretation correta. % salão é **derivado** (`100 - prof_pct`); CHECK schema-level (0-100) garante o range; UI mostra derivação visualmente em tempo real ao lado do input. Nenhum campo `salon_percent` separado é criado. Nenhum stored field em `salons.settings_jsonb` precisa casar — a regra é uma identidade trivial (matemática: x + (100 - x) ≡ 100), não uma restrição de banco. Task 3.4 implementa a apresentação visual.
