## Story

**As a** salon owner,
**I want** to receive password recovery emails and notifications reliably (≥ 95% inbox in Gmail/Outlook),
**so that** I don't lose access to the platform and my clients receive appointment confirmations.

## Acceptance Criteria

1. Conta Resend criada; domínio `<dominio-softhair>` verificado via DNS — SPF/DKIM/DMARC passam em mxtoolbox.com
2. SMTP custom aplicado em `softhair-dev` via Management API (patch `mailer.secure_password_change`, `smtp_*` via script)
3. SMTP custom aplicado em `softhair-prod` via mesma API
4. Script `scripts/supabase-auth-setup.ts` atualizado para incluir campos `smtp_host`, `smtp_port`, `smtp_user`, `smtp_pass`, `smtp_sender_email`, `smtp_sender_name`, `smtp_max_frequency`
5. Secrets (Resend API key) armazenados fora do repo (Vercel env vars OR Supabase Vault); rotação documentada
6. Smoke test em dev: enviar reset password para conta de teste, confirmar chegada em Gmail (inbox, não spam) em ≤ 2 min
7. Smoke test em prod (pós-apply): idem, com Founder validando recebimento real
8. Change Log de `softhair-prod` registrado em `docs/ops/prod-smtp-config-<data>.md`

## Dev Notes

### Context

Esta story é gerada como resultado do `correct-course` de 2026-04-22 — teste em produção do pivot ADR-0003 revelou que built-in SMTP do Supabase tem entrega inadequada (rate limit 429 + IP de baixa reputação). Ver `docs/change-records/2026-04-22-email-delivery-smtp-custom.md` para detalhes completos.

### Vendor choice: Resend

**Rationale:**
- Free tier robusto: 100 emails/dia + 3.000/mês (cobre 20 design-partners do MVP)
- Setup em ≤ 10 min (DNS + API key)
- SMTP endpoint compatível com Supabase sem nada customizado
- Dashboard com analytics de delivery/open/bounce
- Upgrade path: Pro $20/mês = 50.000 emails/mês quando escalar

**Alternativas descartadas:**
- Postmark: melhor reputação de deliverability mas $15/mês mínimo (sem free tier real)
- AWS SES: mais barato em escala mas setup de IAM + sandbox escape é overhead
- SendGrid: reputação de deliverability piorou nos últimos 2 anos

### Management API fields (Supabase Auth)

Reference: https://supabase.com/docs/reference/api/v1-update-auth-service-config

```
PATCH /v1/projects/{ref}/config/auth
{
  "smtp_host": "smtp.resend.com",
  "smtp_port": 587,
  "smtp_user": "resend",
  "smtp_pass": "<api_key>",
  "smtp_sender_email": "noreply@<dominio>",
  "smtp_sender_name": "SoftHair",
  "smtp_max_frequency": 60
}
```

### DNS records (Resend Dashboard → Domains)

Typical pattern (Resend gera os valores reais):
```
TXT  send.<dominio>   "v=spf1 include:amazonses.com ~all"
TXT  resend._domainkey.<dominio>   "p=MIGfMA0GCSqG..."
TXT  _dmarc.<dominio>   "v=DMARC1; p=none;"
```

Propagação DNS pode levar 5–60 min. Validar com `dig TXT resend._domainkey.<dominio>` antes de apply.

### Security — secret management

- `RESEND_API_KEY` NUNCA commitada. Armazenar:
  - Local: `.env.local` (gitignored)
  - Prod apply: variável de ambiente na shell do @devops no momento do apply
  - Supabase Auth armazena a key internamente após apply (não precisa ficar em env do Vercel, pois Supabase é quem envia os emails)
- Rotação: mínimo a cada 6 meses OU se suspeitar de exposição
- Revogação: Resend Dashboard → API Keys → Revoke (imediato)

### Testing

**Smoke test em dev (Task 4):**
1. Conta de teste em softhair-dev (pode ser a mesma `contato.*` já existente em prod; criar homóloga em dev)
2. Trigger manual de reset via UI ou via `supabase.auth.admin.generateLink`
3. Inbox real (Gmail pessoal)
4. Confirmar headers: `dkim=pass`, `spf=pass`, `dmarc=pass` (Gmail mostra ao expandir)

**Smoke test em prod (Task 6):**
- Mesmo, com Founder como receptor

### Out of scope

- Templates customizados adicionais (já cobertos por `RECOVERY_TEMPLATE_PTBR` + `CONFIRMATION_TEMPLATE_PTBR` no script)
- Inbound email (respostas do usuário) — não necessário no MVP
- Multi-vendor fallback — over-engineering para 20 design-partners
- Monitoring de bounce rate — Resend Dashboard cobre; automação vem em Phase 2

### Known gotchas

- Resend free tier: 100 emails/DIA (não mês!) + 3.000/mês. Se disparar onboarding em massa, pode estourar diário
- Domain verification pode falhar se o domínio já tem SPF/DKIM de outro provedor — Resend Dashboard mostra conflito claramente
- `smtp_max_frequency: 60` (seconds) é rate limit POR recipient — usuário não pode receber 2 emails em < 60s
