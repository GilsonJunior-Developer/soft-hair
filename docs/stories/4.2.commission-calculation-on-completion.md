## Story

**As a** sistema,
**I want** calcular comissão automaticamente ao marcar serviço como atendido (transição para `COMPLETED`) e persistir o valor como snapshot imutável,
**so that** o profissional saiba quanto ganhou em cada atendimento sem input manual, e o relatório mensal de Story 4.3 tenha dados confiáveis para fechar a folha de pagamento.

## Acceptance Criteria

> Copiados verbatim de `docs/prd/epic-4-finance-commission-nfs-e.md` linhas 23-31.

1. Ao mudar status para `COMPLETED`, worker calcula comissão aplicando regra vigente
2. Registro persistido em `commission_entries` (agendamento_id, profissional_id, valor_servico, percentual_aplicado, valor_comissao, data_atendimento)
3. Comissão exibida no card do agendamento pós-conclusão
4. Se houver desconto aplicado no serviço, comissão é calculada sobre valor descontado (documentado em Definition of Done)
5. Testes unit cobrem: % fixa, tabela, override, desconto, múltiplos serviços no mesmo agendamento

### CodeRabbit Focus Areas

**Primary Focus**:
- **Idempotency:** if Server Action retries (network failure mid-transaction), the second attempt must NOT create a duplicate `commission_entries` row. UNIQUE constraint on `appointment_id` gives the structural guarantee; code must `INSERT ... ON CONFLICT (appointment_id) DO NOTHING` and treat conflict as success.
- **Discount contract (AC4):** caller MUST pass `appointments.price_brl_final` (post-discount) to `calculateCommission`, NOT `price_brl_original`. Mis-wiring would silently inflate commissions.
- **AC5 immutability invariant — first writer to `commission_entries`:** Story 4.2 is the inaugural insert site. Subsequent code MUST NEVER UPDATE this table. Verify with `grep -r 'UPDATE.*commission_entries' apps/web/` (still 0 matches after this story).
- **Engine consumer pattern:** import canonical `resolveRate` + `calculateCommission` from `@/lib/commission/`. DO NOT duplicate the math (existing 4.1-MNT-001 already flags this anti-pattern in `commission-table-panel.tsx`).

**Secondary Focus**:
- **Read path consistency:** when fetching for display, prefer the snapshot in `commission_entries.commission_amount_brl` over recomputing from current rule (rules may have changed; snapshot is source of truth post-COMPLETED).
- **Error boundaries:** if engine throws (defensive — shouldn't happen with valid inputs), the status transition itself must NOT roll back. The transition is the source of truth; commission insert failure is recoverable (background job could backfill).

### Architectural decision: Server Action as commission writer (not RPC extension)

PRD AC1 says "worker calcula". Two interpretations:
- **A) DB trigger or RPC extension** — atomic, server-authoritative, but duplicates JS engine in plpgsql.
- **B) Server Action post-RPC** — non-atomic, but reuses canonical JS engine.

**Decision: Option B.** Rationales:
1. Engine in JS is testable (vitest with no DB mocks). Plpgsql commission engine would be untestable without a live DB.
2. AC5 immutability is satisfied either way (the engine is pure regardless of language; the invariant is "never UPDATE existing rows", not "atomicity").
3. Idempotency from `UNIQUE(appointment_id)` + `ON CONFLICT DO NOTHING` covers the retry scenario sufficiently.
4. Failure mode: status transition succeeds, commission insert fails → recoverable via future reconciliation. The transition is the source of truth; commission is derived data.
5. Avoids violating DRY against canonical engine — `4.1-MNT-001` already flagged this anti-pattern in the panel UI.

PRD's "worker" terminology was likely planning vocabulary, not architectural mandate. Synchronous Server Action satisfies the user-visible contract: "automatic, no manual input".

### AC2 `data_atendimento` mapping (resolved per @po validation)

PRD AC2 lists 6 named fields including `data_atendimento` (date of attendance). The 6-field mapping resolves as:

| PRD field | Schema source |
|---|---|
| `agendamento_id` | `commission_entries.appointment_id` |
| `profissional_id` | `commission_entries.professional_id` |
| `valor_servico` | `commission_entries.service_price_brl` |
| `percentual_aplicado` | `commission_entries.percent_applied` |
| `valor_comissao` | `commission_entries.commission_amount_brl` |
| `data_atendimento` | **`appointments.scheduled_at` via JOIN** (not denormalized) |

**Why JOIN, not new column:** `commission_entries.created_at` would record "when commission was calculated" (= time of COMPLETED transition), which can drift from the actual appointment time if staff marks COMPLETED late (e.g., 2 days after the service was rendered). Story 4.3 (monthly commission report) will need the JOIN to `appointments` anyway for client name + service name. Adding a redundant `data_atendimento` column would denormalize without read benefit.

**Implication for Task 1.4:** the INSERT does NOT need to capture `data_atendimento` separately. Schema's existing `created_at` (DEFAULT NOW()) records the calc time for audit; the canonical "service date" comes from the JOIN to `appointments.scheduled_at` whenever queries need it.
