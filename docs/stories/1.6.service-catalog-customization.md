## Story

**As a** salon owner,
**I want** to customize the service catalog (add, remove, adjust price and duration),
**so that** the salon reflects its operational reality.

## Acceptance Criteria

1. "Serviços" screen lists all customized services of the salon (seed from onboarding pre-loaded)
2. Add service: from global catalog (`service_catalog`) OR custom creation
3. Each service has: name, category, price (R$), duration (minutes, multiples of 15), custom commission (optional, overrides professional default)
4. Activate/deactivate service (inactive does not appear on public link)
5. Search + filter by category
6. Passes E2E test of service CRUD

### Schema reference (from architecture.md §Data Models)

```
services (
  id UUID PK,
  salon_id UUID FK → salons,
  catalog_id UUID FK → service_catalog (nullable),
  name, category,
  duration_minutes INT CHECK (> 0 AND % 15 == 0),
  price_brl NUMERIC(10,2) CHECK (>= 0),
  commission_override_percent NUMERIC(5,2) CHECK (0-100),
  is_active BOOL,
  deleted_at TIMESTAMPTZ
)
```

### Catalog vs custom

- `catalog_id` IS NOT NULL → service derived from global catalog (editable price, not name/duration)
- `catalog_id` IS NULL → fully custom (all fields editable)
- On delete of catalog entry (superadmin rare action), customer's service survives (FK ON DELETE SET NULL)

### Implementation Notes

1. **Catalog vs custom distinction**: Services derived from `service_catalog` (catalog_id IS NOT NULL) have name/category/duration locked in edit mode. Only price + commission override are editable. Users see an informational banner explaining this and are offered the option to duplicate as a custom service (pending future enhancement).

2. **Commission override opt-in**: Form uses a checkbox to enable commission override. When unchecked, `commission_override_percent` is sent as `null`, meaning the service falls back to the professional's default commission.

3. **Search/filter without client JS**: Implemented as a GET form with `q` and `category` query params. This keeps the page a Server Component and works without hydration. Trade-off: no debounce, requires submit. Acceptable for MVP scale.

4. **Price format**: BRL-friendly (R$ with comma decimal). `formatPrice`/`parsePrice` helpers handle conversion between UI string ("80,00") and numeric value (80.00).

5. **Dedup in catalog picker**: `add-from-catalog-button.tsx` filters out items whose `catalog_id` already exists in salon's active services. The `addServicesFromCatalog` action additionally dedups server-side as defense-in-depth.

6. **Soft delete**: Sets both `deleted_at` and `is_active = false` so the record disappears from both "active" and "inactive" lists.
