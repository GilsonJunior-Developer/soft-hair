## Story

**As a** new salon owner,
**I want** a quick signup wizard (≤ 10 min),
**so that** I start using SoftHair without configuring everything from scratch.

## Acceptance Criteria

1. Wizard in 3 steps: (a) salon data (name, city, CNPJ optional); (b) selection of service categories (hair, nail, barber, aesthetics) — filters standard catalog; (c) prices of 10 most common services per selected category
2. Pre-populated catalog: on selecting category, system pre-adds standard services with suggested prices (editable)
3. Progress bar shows 1/3, 2/3, 3/3
4. "Skip and configure later" button on each step (allows minimum setup with only salon name)
5. On completion, user is redirected to Dashboard "Hoje" (even if empty)
6. Data persists correctly in `salons`, `services` and relations
7. Passes E2E test of complete onboarding in ≤ 10 min

## Dev Notes

### Prerequisite

Story 1.3 (auth) must be Done before this story starts — onboarding requires authenticated user.

### UX (from front-end-spec.md §User Flows §Flow 1)

Sequence diagram in `docs/front-end-spec.md` §Flow 1. Key points:
- 3 steps, progress bar visible
- "Skip" available every step
- Form validates progressively (phone format, CNPJ format)
- Inline errors with `aria-invalid`
- Submit buttons disabled while loading

### RLS bootstrap flow (HIGH-1 + HIGH-3 fixes)

The `salons_insert` + `salon_members_insert` + `salons_select` policies have bootstrap fallbacks for exactly this flow. Verify with Task 5.4 transaction.

### Data transaction strategy

Supabase client doesn't expose explicit transactions from JS SDK. Options:
1. **Postgres function** (RPC): `CREATE FUNCTION create_salon_bootstrap(...) RETURNS salons` — atomic, SECURITY DEFINER
2. **Sequential + compensating rollback** in action: if salon_members insert fails, delete the salon

**Recommendation**: option 1 (RPC). Create migration `20260420000005_create_salon_bootstrap_rpc.sql` in this story.

### Session state for wizard progress

Use `cookies()` from `next/headers` OR client-side React state (if wizard is single-page with tabs). For simplicity and resilience to refresh: use **cookies** scoped `sh_wizard` storing JSON `{ step, categories, ... }`.

### Category → catalog entries mapping

Seed has placeholder catalog (20 entries). For each category:
- Cabelo: 7 entries (from seed)
- Unha: 5 entries
- Barba: 2 entries
- Estética: 6 entries

When catalog expands to 200+ (founder TODO), the filter logic stays the same — just show top 10 per category (by `suggested_price_brl` desc or by `name` asc).

### Testing

**Test file location**: `apps/web/e2e/onboarding.spec.ts` + `apps/web/app/onboarding/actions.test.ts`
**Framework**: Vitest (unit) + Playwright (e2e)
**Standards**:
- E2E runs against Supabase local + seed applied
- Measure full flow duration in Playwright (assert `test.info().duration < 600_000` ms)
- A11y: run `axe-core` via `@axe-core/playwright` on each step

### Out of scope

- Editing salon data post-onboarding (handled in Settings — later story)
- Multi-unit (Phase 3)
- Logo upload (Phase 2)
- Custom services not in catalog (Story 1.6)
