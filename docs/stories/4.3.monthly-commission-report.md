## Story

**As a** dono do salão,
**I want** um relatório mensal de comissão por profissional, com expansão para ver atendimentos individuais e exportação em PDF (impressão) e CSV (folha de pagamento),
**so that** eu pague meus profissionais com rapidez e zero retrabalho, eliminando planilha manual e reconciliação à mão entre agenda e comissão.

## Acceptance Criteria

> Copiados verbatim de `docs/prd/epic-4-finance-commission-nfs-e.md` linhas 32-43.

1. Tela "Comissão" com filtro por período (default: mês corrente)
2. Tabela agregada: profissional, total de atendimentos, faturamento gerado, comissão devida
3. Expansão por linha mostra atendimentos individuais (data, cliente, serviço, valor, comissão)
4. Exportação para PDF (formatado para impressão) e CSV (para folha de pagamento)
5. Agregação executa em ≤ 1s para salão com 500 atendimentos/mês

### CodeRabbit Focus Areas

**Primary Focus**:
- **SQL injection in period filter:** Server Action accepts `from`/`to` ISO date strings from URL searchParams. MUST validate via Zod (`z.string().datetime()` or `z.coerce.date()`) before passing to Supabase query. Reject invalid input with friendly error message. Never string-concat into SQL.
- **Snapshot data only — do NOT recompute from current rules:** This story is a READER. Display columns MUST come from `commission_entries.{percent_applied, commission_amount_brl, service_price_brl}` (snapshot at COMPLETED time), NEVER from current `professionals.commission_default_percent` (which may have changed post-COMPLETED). Mis-wiring would silently drift the report from the immutable history Story 4.2 worked to preserve. AC5 immutability test from 4.2-TEST-001 is the regression sentinel.
- **PostgREST embedding ban (`feedback_no_postgrest_embed.md`):** zero `!inner(...)` / `!left(...)` syntax. Aggregation Server Action does the JOIN in SQL via raw RPC OR fetches `commission_entries` aggregated then resolves professional names via separate `IN` query (precedent: `fetchAgendaWindow` lines 110-126 in `agenda/actions.ts`). No PostgREST embed.
- **CSV format for Brazilian payroll software:** separator `;` (NOT `,` — Excel-BR locale), UTF-8 BOM prefix `﻿` (Excel can't auto-detect UTF-8 without it), CRLF line endings, decimal comma in BRL columns (`R$ 1.234,56`), date format `dd/mm/yyyy`. Without these: Excel breaks accents and reads values as text. Pattern: `[﻿, ...lines.join('\r\n')]` then `Blob` with `text/csv;charset=utf-8`.

**Secondary Focus**:
- **Currency formatting consistency:** if 4.1-MNT-001 + 4.2-MNT-001 cleanup hasn't shipped yet, this story is the **3rd usage of `formatBrl`** — extract to `apps/web/lib/format.ts` per backlog plan (3rd-usage threshold met). Track follow-up if user defers extraction here.
- **Table a11y:** semantic `<table>` with `<thead>` + `<tbody>` + scope on `<th>`. Expandable rows: `aria-expanded` + `aria-controls` on the trigger button + `aria-labelledby` on the disclosure region. Print stylesheet hides interactive controls (filter, expand toggles, export buttons) — show static expanded state of all rows during print.
- **RLS scoping:** existing `commission_entries_select` policy (migration `20260420000002_rls_policies.sql:223-226`) restricts read to user's salon. Server Action runs with user session — RLS enforces automatically. NO service role escalation in this story.
- **Empty-state UX:** salon with zero COMPLETED appointments in selected period must show a clear empty state with the period name, NOT a confusing zero-row table. Pattern precedent: `apps/web/app/(dashboard)/clientes/page.tsx` empty state.

### NO PostgREST embed (codebase convention — non-negotiable)

Established 2026-05-02 via Story 4.2 hotfix PR #33 + reinforced by `feedback_no_postgrest_embed.md` memory. **Never** use `from('X').select('..., Y!inner(...)')` syntax — silently produces wrong shape in production while passing mock tests. Pattern instead:

```typescript
// 1. Fetch primary
const { data: entries } = await sb.from('commission_entries').select('*');

// 2. Resolve relations in parallel via IN
const proIds = [...new Set(entries.map(e => e.professional_id))];
const [proRes, /* ... */] = await Promise.all([
  sb.from('professionals').select('id, name').in('id', proIds),
  // ...
]);

// 3. Map onto entries
const proById = new Map(proRes.data!.map(p => [p.id, p]));
const enriched = entries.map(e => ({ ...e, professionalName: proById.get(e.professional_id)?.name }));
```

Apply to BOTH `fetchCommissionReportSummary` (Task 1.6) AND `fetchCommissionReportRows` (Task 2.3).

### Period scoping decision (Task 1.5)

Two candidate columns to filter by:

| Column | Semantics | Pro | Con |
|---|---|---|---|
| `commission_entries.created_at` | When commission was calculated (= when appointment marked COMPLETED) | Matches "month I'm closing payroll for"; idempotent across appointment date edits; INDEX `idx_commission_entries_professional` covers it directly | Late-marked appointments (e.g., service rendered May 31, marked COMPLETED June 2) appear in June report not May |
| `appointments.scheduled_at` (via JOIN) | When the service was actually performed | Strictly accurate to "service month" | Requires JOIN; needs index on `commission_entries.appointment_id` (already FK + UNIQUE so covered); appointment edits could shift past report retroactively |

**SM recommendation:** start with `created_at` (Task 1.5 default). Document in ADR-0004 amendment. Easy to add `scheduled_at` toggle as a Phase 2 hardening if a partner reports payroll surprise.

### Performance contract (AC5)

500 atendimentos × ~5 indexed queries (one aggregate + one professional-name lookup + drill-downs lazy on click) = O(log n) per query × 5 = ~50ms expected. Existing index `idx_commission_entries_professional` (`salon_id, professional_id, created_at DESC`) is the right shape for the aggregate query. Drill-down query benefits from same index when filtering by `(professional_id, created_at)`.

If Task 8 perf test surfaces >500ms at 50-row scale → escalate to @data-engineer for index audit BEFORE merge.
