## Story

**As a** receptionist,
**I want** to change the appointment status (confirmed, attended, no-show, canceled),
**so that** operational events are tracked and I get no-show metrics for future operational decisions.

## Acceptance Criteria

1. States: `PENDING_CONFIRMATION`, `CONFIRMED`, `COMPLETED`, `NO_SHOW`, `CANCELED` (already defined in schema enum)
2. Valid transitions (state machine, enforced DB + server action):
   - `PENDING_CONFIRMATION` → `CONFIRMED` | `CANCELED`
   - `CONFIRMED` → `COMPLETED` | `NO_SHOW` | `CANCELED`
   - terminal states: `COMPLETED`, `NO_SHOW`, `CANCELED` (no outbound transitions)
3. Actions available from the appointment detail modal (from Story 2.1): "Confirmar", "Marcar atendido", "Marcar no-show", "Cancelar" — only actions valid for the current status are enabled
4. Cancellation opens a small dialog asking optional reason (dropdown: "cliente desistiu", "remarcado", "erro de cadastro", "outro" + free text if "outro")
5. Every transition inserts a row into `appointment_status_log(appointment_id, from_status, to_status, actor_user_id, reason, created_at)` for audit — already enforced by existing trigger (confirm)
6. No-show rate computable over a period: `COUNT(NO_SHOW) / (COUNT(NO_SHOW) + COUNT(COMPLETED))` — query documented, aggregated view optional for MVP
7. Passes E2E test of each transition (deferred — Playwright Phase 2)

### DB trigger contract

**Trigger confirmado existente** (pre-flight @data-engineer 2026-04-21):

```sql
-- from 20260420000003_functions_and_triggers.sql
CREATE TRIGGER appointments_status_log
  AFTER INSERT OR UPDATE OF status ON public.appointments
  FOR EACH ROW EXECUTE FUNCTION public.log_appointment_status_change();
```

Nome real: `appointments_status_log` (não `_ins`). Trigger captura `from_status/to_status/changed_by` via `auth.uid()`, mas NÃO captura `reason`. Solução para cancels com reason → RPC wrapper (Task 2.5).

### Implementation Notes

1. **Enforcement dual-layer:** Server Action valida via Zod enum + state machine helper; RPC valida de novo em SQL (`v_allowed := CASE...`). Impossível ter transição inválida mesmo com cliente malicioso bypassando JS.
2. **Reason capture:** como o trigger existente `appointments_status_log` já dispara no UPDATE e insere a row sem reason, a RPC faz UPDATE adicional no log para setar reason da linha mais recente (mesma tx). Sem duplicata.
3. **UX:** Cancel é a única transição que pede confirmation dialog; outras (Confirmar/Atender/No-show) disparam direto com toast implícito via `router.refresh()`.
4. **Terminal states:** `COMPLETED`, `NO_SHOW`, `CANCELED` não têm outgoing transitions — UI mostra mensagem "Estado final" ao invés de actions.
