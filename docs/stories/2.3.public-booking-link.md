## Story

**As a** salon owner,
**I want** a unique shareable public link for each professional,
**so that** clients can book via Instagram/WhatsApp bio/stories without any friction.

## Acceptance Criteria

1. Each active professional has canonical URL `soft-hair-web.vercel.app/{salon-slug}/{professional-slug}` (in prod, `softhair.com.br` after custom domain)
2. Slug auto-generated from name (lowercase, kebab-case, pt-BR diacritic-stripped, dedup with `-2`/`-3` suffix) — already implemented for `salons` via `slugify()`; apply the same for `professionals`
3. `/profissionais` screen shows "Copiar link público" button per professional card (copies to clipboard, shows "Copiado!" toast)
4. Public page renders SSR (Server Component) with: professional photo (if set, else initials avatar), name, bio, list of services (name + duration + price), salon context (salon name + city), primary CTA "Agendar agora" (handoff to Story 2.4)
5. Page loads in ≤ 2s on 4G (Lighthouse Performance ≥ 90 mobile on the preview deploy)
6. Open Graph + Twitter Card metadata configured for rich preview in WhatsApp/Instagram (og:title, og:description, og:image using the professional photo OR a generated placeholder)
7. Page is WCAG AA compliant (axe-core 0 violations)

## Dev Notes

### Prerequisite

- None beyond Epic 1. Slug for `salons` already works; `professionals.slug` column exists (`initial_schema`), just not enforced unique-per-salon yet (Task 1.1).

### URL shape decision

PRD says `/{salon-slug}/{professional-slug}`. Next.js conflict: `/login`, `/hoje`, etc. are top-level. Route group `(public)` with catch patterns works because Next.js matches more specific routes first. Verify the order:

```
app/
  (auth)/login/page.tsx             → /login
  (dashboard)/hoje/page.tsx         → /hoje
  (public)/[salon]/[professional]/  → /{any}/{any}  ← fallback
```

If Next.js starts matching `/hoje` to the public catch, we'd have a bug. It shouldn't — Next.js App Router gives literal segments higher priority than dynamic ones, and route groups preserve that. Add a test to be safe.

### Public RLS challenge

Supabase default RLS applies to all roles including `anon`. We need:
- `anon` can SELECT from the view with only public columns
- `authenticated` (logged-in user, whether member or not) can also read public views
- No direct SELECT on base tables for `anon`

Using VIEWs with `security_invoker = false` (default) + `GRANT SELECT TO anon` is the cleanest.
