## Story

**As a** salon client (end user),
**I want** to cancel or reschedule my appointment via a unique link,
**so that** I don't need to call the salon.

## Acceptance Criteria

1. Client receives unique URL `soft-hair-web.vercel.app/agendamento/{token}` **via email** after booking confirmation (handoff from Story 2.4 Task 8)
2. Token is a JWT signed with the server secret, with `exp` = appointment `scheduled_at + 4h` (access window closes after appointment ends)
3. Page shows appointment summary + actions "Cancelar" and "Reagendar"
4. Cancellation window configurable by the salon (column `salons.cancel_window_hours` — default 24h). Attempting to cancel inside the window blocks with clear message "Cancelamentos devem ser feitos com {N}h de antecedência"
5. Reschedule opens the same flow as Story 2.4 (new slot selection inside availability)
6. Cancellation updates appointment status to `CANCELED` and frees the slot in real time on the salon agenda (Story 2.1 Realtime subscription)
7. Post-action screen confirms to the client + (if canceled) suggests rescheduling

### Debug Log References

- jose v6 quirks: `jose@6.2.2` webapi path requires `CryptoKey` from `crypto.subtle.importKey('raw', ...)`, not `Uint8Array`. Initial attempt with `new TextEncoder().encode(secret)` failed with `TypeError: payload must be an instance of Uint8Array` inside `FlattenedSign`. Fixed with `importKey('raw', keyData, { name: 'HMAC', hash: 'SHA-256' }, false, ['sign', 'verify'])` + per-secret cache.
- vitest env collision: `lib/appointment-token.test.ts` needed `// @vitest-environment node` override — the default `jsdom` env has a partial WebCrypto implementation that breaks jose's internal sign path. Other tests in the repo stay on jsdom for React Testing Library compatibility.
- `import 'server-only'` dropped from `appointment-token.ts` because vitest cannot resolve the module. The env check in `getSecret()` (no `NEXT_PUBLIC_` prefix) is the effective guard — any client-side import crashes loudly on the missing secret.
- `.next` cache needed manual `rm -rf` after deleting the retired `/book/success` route; typecheck otherwise tripped on stale generated type files.

### Dependência cruzada: resolve 2.4-SEC-001 (docs/qa/backlog.md)

**Requisito obrigatório**, não efeito colateral. A 2.4 foi shipada com um `cancel_token` hex exposto no URL de sucesso via searchParam `?t=...`. Quem chamar `get_public_booking(id, token)` com esse par consegue ler `client_name + client_email` — se o cliente compartilhar a URL publicamente (screenshot em WhatsApp, print em redes sociais), vaza PII de si mesmo. A 2.7 **substitui** esse scheme por JWT signed em path (`/agendamento/{jwt}`), que:
- É assinado com secret server-only (HS256) — tampering invalida
- Tem `exp` atrelado ao `scheduled_at + 4h` — janela curta
- É trocado a cada reschedule (Task 6.4) — não é o mesmo token cruzando tempo

**Critério de fechamento do SEC-001:** após merge da 2.7, o URL `/book/success/[id]?t=...` da 2.4 deve ser aposentado OU o `get_public_booking` RPC deve ter sua GRANT revogada de `anon` (substituído por um RPC token-gated equivalente). @qa Quinn marca 2.4-SEC-001 como `done` no backlog quando confirmar via smoke.

### Why JWT over opaque token

- Self-validating: no DB lookup for the common case (just verify signature + exp)
- Revocation: if we need to invalidate, add a `token_version` column on `appointments` and include in JWT; mismatch = revoked (future)
- Easy to rotate: bump `APPOINTMENT_TOKEN_SECRET` invalidates all existing links (last resort)

### RLS consideration

Public routes run as `anon`. The page needs to read a specific appointment by id from the token. Options:
1. Use `SECURITY DEFINER` RPC `get_appointment_by_token(token)` that verifies internally — safest
2. Add RLS policy allowing `anon` SELECT WHERE id matches a claim — requires custom JWT verification in Postgres (complex)

**Recommended**: Option 1 (RPC). Bypasses RLS safely after token verification.

### Email fallback (if Resend not configured)

Same as Story 2.4: log + skip + show a banner telling the salon to send the link manually. Not ideal but keeps the flow working for design-partner #1 even without email infra.
