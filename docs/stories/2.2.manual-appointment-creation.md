## Story

**As a** receptionist,
**I want** to create an appointment manually when a client calls or walks in,
**so that** no booking is lost regardless of channel.

## Acceptance Criteria

1. "+ Novo agendamento" button visible on every `/agenda/*` view (toolbar) and on `/hoje` empty state
2. Form: cliente (search by phone — autocomplete: if exists, loads; if not, inline "Criar novo cliente" with name+phone+email?), serviço (dropdown of active services filtered by professional catalog), profissional (dropdown with availability filtering), data & hora picker (15-min steps)
3. System auto-computes `ends_at` from `scheduled_at + service.duration_minutes` (enforced by DB trigger already; UI must mirror for preview)
4. Validation: **blocks submit** if conflict exists with another appointment for the same professional in the time range — check via Server Action before insert
5. On confirm, appointment appears immediately in the agenda (via Realtime subscription from 2.1, no page reload)
6. If client is new, row is inserted into `clients` with phone as unique identifier (`phone_e164` UNIQUE per salon) — handled atomically in the same Server Action
7. Passes E2E test of complete creation flow (deferred — Playwright env Phase 2)

## Dev Notes

### Prerequisite

- Story 2.1 must be done (or at least merged) for the agenda view + Realtime hook to consume the new row.

### Conflict-check performance

With the existing `appointments_professional_id_scheduled_at_idx`, a `tstzrange && tstzrange` check on the same professional within a day is cheap (index range scan). For week windows with many pros, still fast.

### Client phone as identifier

`clients.phone_e164` is `UNIQUE (salon_id, phone_e164)` (confirm in initial_schema). Same phone in different salons = different rows (intentional — each salon owns its client list).

### RLS

Staff context has `current_user_salon_ids()` returning `{ salon_id }`. Insert policy on `appointments` checks `salon_id IN ( … )`. No additional tweaks needed.

### Out of scope

- Recurring appointments — Phase 2
- Bulk/overbooking — Phase 2
- Notification to client on creation — comes in Story 2.7 (email link)

### Implementation Notes

3. **Entry points:** toolbar (`/agenda` "Novo agendamento"), `/hoje` empty state CTA, click em slot vazio no day view (pré-preenche horário).
