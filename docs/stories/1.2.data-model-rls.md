## Story

**As a** founder/dev,
**I want** the core data model (salon, user, professional, service, client, appointment) applied with RLS configured,
**so that** multi-tenancy is guaranteed at the database level from the start.

## Acceptance Criteria

1. Supabase schema created with tables: `salons`, `users`, `professionals`, `services`, `clients`, `appointments`, `service_catalog` (global)
2. All domain tables have `salon_id` column (UUID, FK, NOT NULL)
3. RLS policies activated: user only reads/writes rows from their `salon_id`
4. Policy explicitly allows access to the global `service_catalog` (read-only public)
5. Migrations versioned in `packages/db/migrations/`
6. TypeScript types auto-generated via Supabase CLI (`packages/db/types`)
7. RLS tests: unit test validating that user of salon A cannot access salon B's data
8. Seed SQL with 20+ standard services populating `service_catalog`

## Dev Notes

### Pre-existing artifacts

All SQL already written by @data-engineer (see `packages/db/`):

- `migrations/20260420000001_indexes.sql` — 30+ indexes

- `seed.sql` — 20 services + 6 WA templates
- `rollback/20260420000000_rollback.sql` — emergency rollback

This story does NOT write new SQL. It applies + validates what exists.

### RLS security invariants (from security-audit-2026-04-20.md — HIGH fixes applied)

1. **Helper**: `current_user_salon_ids()` returns set of salon_ids where user is member (SECURITY DEFINER)
2. **Pattern**: every domain table `USING (salon_id IN (SELECT current_user_salon_ids()))`
3. **Bootstrap fix**: `salon_members_insert` + `salons_select` have `owner_user_id` fallback for signup flow
4. **Cross-salon FK validation**: triggers in `appointments` + `referrals` block cross-tenant inserts

### Types import strategy

- Canonical types: `packages/db/types/database.ts` (from local Supabase)
- `apps/web` imports via workspace: `import type { Database } from '@softhair/db';`
- Regenerate after EVERY migration: add `db:types` script to `packages/db/package.json`

### Migration application order (CRITICAL)

1. `20260420000000_initial_schema.sql` — tables + enums (foundational)
2. `20260420000001_indexes.sql` — indexes (depends on tables)
3. `20260420000002_rls_policies.sql` — RLS (depends on tables + helper functions reference)
4. `20260420000003_functions_and_triggers.sql` — triggers (depends on tables + functions)

Supabase CLI respects filename order. DO NOT rename files.

- Founder must have created 3 Supabase projects (Story 1.1 Task 3)
- Founder must have credentials in `.env.local` + `.env.example` placeholders
- Database passwords stored in password manager (not committed)

### Testing

**Test file location**: `packages/db/tests/*.test.ts`
**Framework**: Vitest 2.x
**Approach**:
- Each test starts/resets Supabase local via fixture (`beforeAll` starts clean state)
- Use `@supabase/supabase-js` client with anon + service role keys
- Test actual RLS behavior (not mocks) — defense in depth

**Required tests (minimum)**:
- `rls-smoke.test.ts` covering AC 7 (positive + negative scenarios)
- `seed.test.ts` validating seed inserts without error
