### Tailwind v4 specifics

- Uses `@theme` directive in CSS (not config file)
- Tokens from `docs/front-end-spec.md` ┬зDesign System Foundations should be added to `apps/web/app/globals.css` using CSS custom properties (will be formalized in design-system migration but this story just needs Tailwind v4 installed and baseline working)

### Supabase RLS constraint

- In CI, DO NOT connect to live Supabase тАФ all unit tests must use Supabase local (via `supabase start`) or mocked client
- Service role key NEVER goes to client bundle тАФ verify via Next.js `env` scoping (`NEXT_PUBLIC_*` vs server-only)

### Testing

**Test file location**: co-located with source (`*.test.ts` next to `*.ts`)
**Framework**: Vitest 2.x for unit/integration; Playwright for e2e (not in this story)
**Standards**:
- Each Server Action has at least 1 happy-path + 1 error-path test
- All domain logic in `packages/core` has тЙе90% line coverage
- Use `msw` or equivalent for mocking external APIs

**For this story specifically**:
- Test the `/healthz` route (simplest smoke test)
- No other tests required тАФ scaffold validation only

### Out of scope for this story (do NOT implement)

- Auth (Story 1.3)
- Data model / migrations (Story 1.2 тАФ migrations already written in `packages/db/migrations/`)
- Onboarding wizard (Story 1.4)
- Any UI beyond `/` (homepage placeholder) and `/api/healthz`
