## Story

**As a** salon owner,
**I want** to see a clear initial screen showing what will happen today,
**so that** I have immediate context when opening the system.

## Acceptance Criteria

1. Dashboard displays: today's date, next 3 appointments (empty now, friendly placeholder "Nenhum agendamento hoje ainda"), 3 placeholder metrics (appointments today, daily revenue, occupancy rate — all zero)
2. Header with logo, salon name, user avatar + logout menu
3. Lateral navigation (desktop) or bottom-nav (mobile) with: Hoje, Agenda, Profissionais, Serviços, Clientes (placeholder), Financeiro (placeholder), Configurações
4. Responsive mobile-first: layout adapts at ≤ 375px
5. PWA manifest configured (installable)
6. Accessibility: keyboard navigation + ARIA labels validated with axe-core
7. Passes E2E rendering test in Chrome + Safari iOS (via Playwright)

### PWA gotchas

- iOS Safari: requires `apple-mobile-web-app-capable` meta + specific icon sizes
- Android Chrome: install prompt controlled by `beforeinstallprompt` event — can be custom UI (front-end-spec mentions custom prompt after 3 visits)
- Service worker: start simple (network-first for pages, cache-first for assets); can optimize in Phase 2

### Font loading strategy (Tailwind v4 + next/font)

```tsx
// apps/web/app/layout.tsx
import { Inter, Playfair_Display } from 'next/font/google';

const inter = Inter({ subsets: ['latin'], variable: '--font-sans' });
const playfair = Playfair_Display({ subsets: ['latin'], variable: '--font-display' });

export default function RootLayout({ children }) {
  return (
    <html lang="pt-BR" className={`${inter.variable} ${playfair.variable}`}>
      <body>{children}</body>
    </html>
  );
}
```

### Implementation Notes

1. **Responsividade sem JS**: Sidebar e BottomNav usam apenas `hidden lg:flex` / `lg:hidden` — alternância de breakpoint é CSS puro. Zero hydration mismatch, zero JS para detectar viewport.

2. **Active state robusto**: Cálculo é `pathname === href || pathname.startsWith(href + "/")` para marcar o item correto quando em rotas aninhadas (ex.: `/profissionais/abc-123` destaca "Profissionais").

3. **Accessibility (AC 6)**:
   - Todos os nav têm `aria-label` descritivo e `aria-current="page"` no item ativo
   - UserMenu usa `aria-haspopup`, `aria-expanded`, click-outside via mousedown listener, Esc fecha o menu
   - BottomNav sheet usa `role="dialog"` + `aria-modal` + botão de fechar acessível
   - Focus ring global já definido em `globals.css` com `:focus-visible` de 2px accent-500
   - Icons sempre com `aria-hidden` (acompanhados de texto)
   - O auditoria formal via axe-core fica para Task 8 (deferida)

4. **Safe-area iOS**: BottomNav aplica `env(safe-area-inset-bottom, 0)` no padding-bottom para não colidir com home indicator. Viewport usa `viewportFit: cover`. Layout do main tem `pb-24 lg:pb-8` para não ficar atrás do BottomNav.

5. **PWA icons SVG**: Usados SVG em vez de PNG porque: (a) não precisam de build tool de imagem, (b) escalam perfeitamente, (c) SoftHair logo final ainda não está definido — placeholder "SH" é fácil de substituir. Android Chrome e iOS Safari (17+) aceitam SVG em manifest. Caso um dispositivo antigo recuse, o install prompt ainda funciona (usa fallback).

6. **Service worker deferido**: Next.js 15 App Router já oferece prefetching agressivo, RSC streaming e cache robusto. Instalar `next-pwa` ou registrar SW customizado agora adicionaria complexidade (build config, update strategy, cache invalidation) sem ganho claro para MVP sem ainda ter uso offline crítico. Task fica explicitamente marcada para Phase 2.

7. **Greeting sem `user.name`**: O user table não tem `full_name` neste MVP — deduzo o nome do local-part do e-mail (`camila@tely.com.br` → "Camila"). Se o valor for vazio, usa "por aqui" como fallback silencioso. Pode evoluir quando houver campo de nome no profile.
