## Story

**As a** salon owner,
**I want** to sign in using only my WhatsApp number (no password),
**so that** access is simple and frictionless, aligned with natural behavior of the beauty sector.

## Acceptance Criteria

1. Login screen accepts WhatsApp number (BR format: +55 DDD 9XXXX-XXXX)
2. On submit, system generates 6-digit OTP valid for 10 minutes
3. OTP is sent via WhatsApp BSP (Meta Cloud API — ADR-0001) using approved utility template
4. OTP validation screen accepts code and returns valid Supabase Auth session
5. Session expires after 30 days of inactivity
6. Rate limit: max 3 OTP requests per phone per 10 minutes
7. Logout clears session and redirects to login screen
8. Clear fallback if BSP fails (error message + "try again in X seconds")
9. Full flow passes E2E test (Playwright) using BSP mock

### CodeRabbit Focus Areas

**Primary Focus**:
- OTP generation: use cryptographically secure random (`crypto.randomInt`), not `Math.random`
- Constant-time comparison for OTP validation (timing attack prevention)
- Rate limit: atomic Redis INCR or Postgres advisory lock (no race window)

**Secondary Focus**:
- Error messages: do NOT leak whether phone is registered ("If your number is registered, we sent a code")
- Logs: never include OTP value; only hashed reference

### Tables added by this story (migration 20260420000004)

```sql
CREATE TABLE public.auth_otps (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  phone_e164 CITEXT NOT NULL,
  otp_hash TEXT NOT NULL,
  expires_at TIMESTAMPTZ NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_auth_otps_phone_expires ON auth_otps (phone_e164, expires_at DESC);

CREATE TABLE public.auth_otp_attempts (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  phone_e164 CITEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_auth_otp_attempts_phone_time ON auth_otp_attempts (phone_e164, created_at DESC);
```

Both tables have RLS ENABLED with zero policies — accessed only via service_role in Route Handlers.

Cron cleanup: Inngest job in Story 3.x (cleanup expired rows).

### UX (from front-end-spec.md)

### Security constraints

1. OTP generated via `crypto.randomInt` — never Math.random
2. OTP comparison via bcrypt (constant-time)
3. Rate limit enforced BEFORE OTP generation (prevents flood attacks)
4. Generic error messages (don't leak phone existence)
5. OTP logs: only log phone hash + attempt result, never raw OTP
6. Session cookie: `SameSite=Lax`, `Secure` (prod), `HttpOnly`
